Selbsteinschätzung Developer Model

Wofür diese Seite gedacht ist

Diese Seite ist eine statische Erweiterung des Open-Source-Projekts engineeringladders. Sie macht den Developer-Teil als wiederverwendbares Interview- und Selbstreflexionswerkzeug nutzbar: Werte auswählen, eigene Belege sammeln, das Ergebnis als Spinnennetzdiagramm prüfen und die Einschätzung teilen oder exportieren.

Die grauen Linien zeigen das originale Modell als Referenz. Die rote Fläche zeigt deine aktuelle eigene Einschätzung.

So funktioniert die Nutzung

  • Profil und Stand: Optionaler Name und Datum für Ausdruck, Export und geteilte URL.
  • Regler: Jede Achse kann von 0 bis 5 bewertet werden. 0 bedeutet nicht bewertet.
  • Ermitteltes Level: Unter dem Diagramm wird das höchste Developer-Level D1 bis D7 angezeigt, dessen Achsenanforderungen vollständig erfüllt sind.
  • Belege / Beispiele: Freitext für konkrete Situationen, Projekte oder Verantwortungen.
  • Zusammenfassung: Unter dem Diagramm erscheinen der gewählte Level, die originale Level-Definition und deine Einschätzung.
  • URL teilen: Schreibt den vollständigen Zustand inklusive Notizen in den URL-Hash und kopiert die URL, wenn der Browser es erlaubt.
  • JSON exportieren/importieren: Speichert oder lädt Profil, Datum, Werte und Notizen als Datei.
  • Alles zurücksetzen: Leert Werte, Notizen, Profil, lokale Daten und geteilten URL-Hash.

Zusätzliche Funktionen dieser Version

  • Statische GitHub-Pages-Version: Die Anwendung besteht aus einer HTML-Datei mit CSS und JavaScript und braucht keinen Server.
  • Wiederverwendbare Eingaben: Die Zusammenfassung nutzt deine eigenen Belege statt fest vorbereiteter CV-Texte.
  • PDF-CV-Import: Ein CV kann lokal im Browser analysiert werden. Es wird nichts hochgeladen.
  • Prüfbare CV-Vorschläge: Erkannte Begriffe werden als auswählbare Chips angezeigt und können einzeln entfernt werden, bevor sie übernommen werden.
  • Teilen und Import/Export: Die komplette Einschätzung kann per URL oder JSON weitergegeben werden.
  • Druckansicht: Die Seite blendet Eingabefelder aus und bereitet Diagramm und Zusammenfassung für den Druck vor.

Technische Umsetzung

  • Rendering: Das Diagramm wird als SVG erzeugt. Die Punkte werden aus den fünf Achsen und den Werten 0 bis 5 berechnet.
  • Developer-Level: Die D1-D7-Profile aus Developer.md sind als Achsenanforderungen hinterlegt. Angezeigt wird das höchste Profil, bei dem jede Achse mindestens den geforderten Wert erreicht.
  • Status: Ein JavaScript-State-Objekt hält Profil, Datum, Achsenwerte und Notizen. Nach jeder Änderung werden Diagramm, Eingaben und Karten neu gerendert.
  • Persistenz: Der Zustand wird in localStorage gespeichert. Die geteilte URL kodiert denselben Zustand als JSON im URL-Hash.
  • Sicherheit: Freitexte werden vor der Ausgabe escaped, damit keine HTML-Inhalte aus Textfeldern ausgeführt werden.
  • PDF-Verarbeitung: PDF.js wird lokal aus diesem GitHub-Projekt geladen und liest Text aus textbasierten PDFs direkt im Browser. OCR für gescannte PDFs ist nicht enthalten.
  • CV-Heuristik: Eine transparente Keyword-Liste pro Achse erkennt Begriffe wie Technologien, Betrieb, Mentoring, Prozesse oder Wirkungskreise. Daraus werden Vorschlagswerte, Konfidenz und auswählbare Begriff-Chips gebildet.
  • Kein AI-Backend: Es werden keine API-Keys, Serverfunktionen oder Uploads verwendet. Die Analyse ist deterministisch und lokal.

CV-Import: Parser und Zuordnung

Der CV-Import ist bewusst als lokale, nachvollziehbare Heuristik gebaut. Er soll beim Sortieren und Vorbereiten von Belegen helfen, ersetzt aber keine fachliche Bewertung durch die Person, die das Interview oder die Selbstreflexion vorbereitet.

  • Dateiauswahl: Der Button CV importieren öffnet ein verstecktes PDF-Dateifeld. Die ausgewählte Datei bleibt im Browser und wird nicht an einen Server gesendet.
  • Textauslesung: PDF.js liest jede PDF-Seite mit getTextContent() aus. Die Textfragmente der Seiten werden zusammengeführt.
  • Normalisierung: Der extrahierte Text wird bereinigt: geschützte Leerzeichen werden ersetzt, mehrere Leerzeichen reduziert und überflüssige Zeilenumbrüche geglättet.
  • Grenze für leere PDFs: Wenn zu wenig Text extrahiert wird, wird keine Analyse angezeigt. Das betrifft zum Beispiel gescannte PDFs ohne OCR-Textschicht.
  • Achsen-Mapping: Jede der fünf Achsen hat eine eigene Keyword-Liste: Technology, System, People, Process und Influence.
  • Keyword-Suche: Die Begriffe werden mit regulären Ausdrücken gesucht, damit nicht einfach beliebige Teilstrings gewertet werden. Varianten wie Node.js und nodejs werden anschließend zusammengeführt.
  • Gewichtung: Jedes gefundene Keyword hat ein Gewicht. Stärkere Signale wie SLAs, Kubernetes, Cross-Team oder Community zählen höher als allgemeinere Begriffe.
  • Level-Vorschlag: Die gewichtete Summe einer Achse wird in einen Wert von 0 bis 5 übersetzt. Dieser Wert ist ein Vorschlag, keine automatische Einstufung.
  • Konfidenz: Die Anzeige niedrig, mittel oder hoch basiert auf Score, vorgeschlagenem Level und Anzahl gefundener Textstellen.
  • Begriff-Chips: Gefundene Begriffe werden als auswählbare Chips angezeigt. Nutzer können einzelne Begriffe entfernen, bevor die Vorschläge übernommen werden.
  • Import in die Einschätzung: Erst nach Vorschläge übernehmen werden die verbleibenden Begriffe in die Notizfelder geschrieben und die Regler auf die vorgeschlagenen Werte gesetzt.
  • Transparenz: Es gibt kein verstecktes Modell und keine externe Bewertung. Die Vorschläge entstehen nur aus den sichtbaren Keyword-Listen und der lokalen PDF-Textauslesung.

Geltungsbereich

Diese Version ist aktuell nur für den Developer-Teil des Engineering-Ladders-Modells umgesetzt. Das Originalprojekt enthält weitere Markdown-Varianten wie Tech Lead, Technical Program Manager und Engineering Manager. Die gleiche statische Struktur könnte dafür angepasst werden, wenn die Achsen, Level-Texte und Heuristiken entsprechend ausgetauscht werden.

Developer-Level aus Developer.md

Die folgende Übersicht ist eine deutsche Zusammenfassung der Developer-Level aus Developer.md.

D1 - Developer 1

  • Technology: lernt und nutzt aktiv die vom Team definierten Technologien und Werkzeuge.
  • System: liefert Features und Bugfixes, die das System verbessern und erweitern.
  • People: lernt schnell von anderen und springt verlässlich ein, wenn es nötig ist.
  • Process: folgt den Teamprozessen und liefert kontinuierlich Features in Produktion.
  • Influence: wirkt auf ein oder mehrere Subsysteme oder Team-Pods.

D2 - Developer 2

  • Technology: lernt und nutzt weiterhin aktiv die vom Team definierten Technologien und Werkzeuge.
  • System: designt und implementiert mittlere bis große Features und reduziert technische Schulden.
  • People: unterstützt andere Teammitglieder proaktiv und hilft ihnen erfolgreich zu sein.
  • Process: setzt Teamprozesse durch und erklärt Nutzen und Trade-offs.
  • Influence: wirkt auf ein oder mehrere Subsysteme oder Team-Pods.

D3 - Developer 3

  • Technology: ist Ansprechperson für eine oder mehrere Technologien und lernt eigeninitiativ neue.
  • System: designt und implementiert mittlere bis große Features und reduziert technische Schulden.
  • People: unterstützt andere Teammitglieder proaktiv und hilft ihnen erfolgreich zu sein.
  • Process: hinterfragt Teamprozesse und sucht Verbesserungsmöglichkeiten.
  • Influence: wirkt auf das ganze Team, nicht nur auf einzelne Teile.

D4 - Developer 4

  • Technology: recherchiert, erstellt Proofs of Concept und bringt neue Technologien ins Team.
  • System: verantwortet Betrieb und Monitoring des Systems und kennt dessen SLAs.
  • People: mentort andere, beschleunigt Karrierewachstum und fördert Beteiligung.
  • Process: hinterfragt Teamprozesse und sucht Verbesserungsmöglichkeiten.
  • Influence: wirkt auf das ganze Team, nicht nur auf einzelne Teile.

D5 - Developer 5

  • Technology: kennt den gesamten Technologie-Stack des Systems sehr tief.
  • System: entwickelt die Architektur für künftige Anforderungen weiter und definiert SLAs.
  • People: mentort andere, beschleunigt Karrierewachstum und fördert Beteiligung.
  • Process: passt Teamprozesse anhand von Feedback an und begleitet das Team durch Veränderungen.
  • Influence: wirkt nicht nur im eigenen Team, sondern auch in anderen Teams.

D6 - Developer 6

  • Technology: designt und schafft neue Technologien, die intern oder extern breit genutzt werden.
  • System: führt technische Exzellenz des Systems und plant Maßnahmen gegen Ausfälle.
  • People: mentort andere, beschleunigt Karrierewachstum und fördert Beteiligung.
  • Process: passt Teamprozesse anhand von Feedback an und begleitet das Team durch Veränderungen.
  • Influence: wirkt auf die gesamte Tech-Organisation.

D7 - Developer 7

  • Technology: designt und schafft neue Technologien, die intern oder extern breit genutzt werden.
  • System: führt technische Exzellenz des Systems und plant Maßnahmen gegen Ausfälle.
  • People: mentort andere, beschleunigt Karrierewachstum und fördert Beteiligung.
  • Process: passt Teamprozesse anhand von Feedback an und begleitet das Team durch Veränderungen.
  • Influence: wirkt auf die technische Community.

Datenschutz

Lokale Verarbeitung und gespeicherte Daten

Kurzfassung

Diese Seite ist als statische GitHub-Pages-Anwendung gebaut. Die Selbsteinschätzung, Notizen und der CV-Import laufen im Browser. Es gibt in dieser Version keinen eigenen Server, kein AI-Backend, keine API-Keys und keinen Upload deines CVs durch diese Anwendung.

CV-Import

  • Lokale Datei: Der ausgewählte PDF-CV wird im Browser mit PDF.js gelesen und nicht an diese Website, GitHub oder einen eigenen Backend-Service hochgeladen.
  • Lokale Bibliothek: PDF.js wird aus dem Ordner vendor/pdfjs dieses Projekts geladen. Dadurch entsteht für die PDF-Verarbeitung kein zusätzlicher CDN-Aufruf an einen Drittanbieter.
  • Keine OCR: Gescannte PDFs ohne Textschicht werden nicht ausgelesen. In diesem Fall müssen Belege manuell eingetragen werden.
  • Keine automatische Entscheidung: Die Vorschläge entstehen aus lokalen Keyword-Regeln und müssen aktiv geprüft und übernommen werden.

Lokale Speicherung

  • Browser-Speicher: Profil, Datum, Achsenwerte und Notizen werden im localStorage deines Browsers gespeichert, damit die Eingaben beim Neuladen erhalten bleiben.
  • Zurücksetzen: Alles zurücksetzen löscht diese lokalen Daten, die Notizen, das Profil und den geteilten URL-Hash.
  • JSON: Beim JSON-Export wird eine Datei lokal im Browser erzeugt. Beim JSON-Import wird nur die ausgewählte Datei im Browser gelesen.

Teilen per URL

URL teilen schreibt den Zustand der Einschätzung inklusive Notizen in den URL-Hash. Wer diese URL erhält, kann die enthaltenen Daten lesen. Nutze diese Funktion daher nur, wenn die Notizen für die Empfänger bestimmt sind.

Hosting durch GitHub Pages

Die Website wird über GitHub Pages ausgeliefert. Beim Aufruf einer GitHub-Pages-Seite können technische Zugriffsdaten wie IP-Adresse und Request-Metadaten durch GitHub verarbeitet werden, insbesondere für Sicherheit und Betrieb des Dienstes. Diese Anwendung selbst betreibt keine zusätzliche serverseitige Protokollierung.

Empfehlung

Trage nur Informationen ein, die du für die Selbstreflexion oder das Gespräch verwenden möchtest. Prüfe CV-Vorschläge vor dem Übernehmen und teile URLs oder JSON-Dateien nur bewusst mit den Personen, die diese Inhalte sehen sollen.
